    Congestion Control in Computer Networks

    1. Introduction to Congestion Control

    Congestion control is a fundamental aspect of computer networking that regulates the amount of data entering the network to prevent overload. When too many packets are transmitted through a network segment, congestion occurs, leading to packet loss, increased delay, and reduced throughput.

    Key Definition: Congestion control refers to the mechanisms and techniques used to control the entry of traffic into a telecommunications network to avoid congestive collapse by attempting to avoid oversubscription of any of the processing or link capabilities of intermediate network nodes.

    Congestion control differs from flow control in that:

    • Flow control prevents a fast sender from overwhelming a slow receiver
    • Congestion control prevents too much data from being injected into the network, which would cause routers/switches to become overloaded

    2. Causes of Network Congestion

    Several factors can contribute to network congestion:

    Factor Description
    Bottlenecks When multiple input streams converge on a router whose output capacity is less than the sum of the inputs
    Slow Processors Routers with slow CPUs cannot process packets fast enough
    Low Bandwidth Links Links with insufficient capacity for the traffic load
    Buffer Overflow When router queues fill up and packets must be dropped
    Bursty Traffic Sudden increases in traffic that exceed network capacity

    Effects of Congestion

    Uncontrolled congestion can lead to:

    1. Increased packet delay and jitter
    2. Reduced throughput and network efficiency
    3. Packet loss due to buffer overflow
    4. Unfair allocation of bandwidth among flows
    5. Congestive collapse (complete breakdown of communication)

    3. Principles of Congestion Control

    1. Monitoring

    Detecting when and where congestion occurs by measuring:

    • Packet loss rate
    • Queue length in routers
    • Average delay
    • Throughput

    2. Feedback

    Communicating congestion information:

    • Explicit feedback (ECN, ICMP)
    • Implicit feedback (packet loss, delay)
    • End-to-end vs. hop-by-hop

    3. Rate Adjustment

    Adjusting transmission rates based on feedback:

    • Additive Increase/Multiplicative Decrease
    • Window-based control
    • Rate-based control

    Congestion Control Taxonomy

    1. Open-loop vs. Closed-loop:

    • Open-loop: Prevent congestion before it happens (resource reservation, traffic shaping)
    • Closed-loop: React to congestion after it occurs (TCP congestion control)

    2. Router-based vs. Host-based:

    • Router-based: Active Queue Management (RED, ECN)
    • Host-based: TCP congestion control algorithms

    3. Window-based vs. Rate-based:

    • Window-based: Adjust window size (TCP)
    • Rate-based: Directly control transmission rate (ATM ABR)

    4. Congestion Control Approaches

    4.1 Prevention-based Approaches

    These methods attempt to prevent congestion before it occurs:

    Traffic Shaping

    Regulating the average rate and burstiness of data flow:

    • Leaky Bucket: Fixed output rate
    • Token Bucket: Allows burstiness within limits

    Resource Reservation

    Reserving network resources before transmission:

    • RSVP (Resource Reservation Protocol)
    • IntServ (Integrated Services)

    Admission Control

    Rejecting new flows when network is near capacity:

    • Used in ATM networks
    • Common in QoS architectures

    4.2 Reaction-based Approaches

    These methods detect and respond to congestion:

    Choke Packets

    When a router detects congestion, it sends a choke packet back to the source host, which is required to reduce its transmission rate by a specified percentage.

    Load Shedding

    When routers are overwhelmed, they discard packets according to some policy:

    • Tail drop (simplest approach)
    • Priority-based (e.g., discard less important packets first)
    • Random early discard

    5. TCP Congestion Control

    TCP implements several congestion control algorithms that have evolved over time:

    5.1 TCP Tahoe and Reno

    The original TCP congestion control mechanisms:

    Slow Start

    Exponential growth of congestion window (cwnd) until threshold:

    • Start with cwnd = 1 MSS
    • Double cwnd every RTT
    • Continue until ssthresh or loss occurs

    Congestion Avoidance

    Additive increase after reaching ssthresh:

    • Increase cwnd by 1 MSS per RTT
    • Linear growth instead of exponential

    Fast Retransmit

    When sender receives 3 duplicate ACKs:

    1. Retransmit missing segment immediately
    2. Don't wait for timeout
    3. Reduces latency for lost packets

    Fast Recovery

    After fast retransmit (TCP Reno only):

    1. Set cwnd = ssthresh + 3 MSS
    2. Increase cwnd by 1 MSS per duplicate ACK
    3. Exit when ACK for new data arrives

    5.2 Modern TCP Variants

    Variant Key Features Use Case
    TCP Vegas Uses RTT measurements to predict congestion Low-loss networks
    TCP BBR Model-based, measures bottleneck bandwidth and RTT High-speed, long-distance
    TCP CUBIC Cubic function for window growth, default in Linux High-speed networks
    TCP Westwood Bandwidth estimation to set ssthresh Wireless networks

    6. Active Queue Management (AQM)

    AQM techniques are implemented in routers to manage buffer occupancy and provide early congestion notification:

    6.1 Random Early Detection (RED)

    RED monitors the average queue size and drops (or marks) packets probabilistically:

    RED Algorithm

    1. Compute average queue size (avg)
    2. If avg < min_th: no drop
    3. If min_th ≤ avg < max_th: drop with probability p
    4. If avg ≥ max_th: drop all packets

    RED Parameters

    • min_th: minimum threshold
    • max_th: maximum threshold
    • max_p: maximum drop probability
    • w_q: queue weight factor

    6.2 Explicit Congestion Notification (ECN)

    ECN allows routers to mark packets instead of dropping them when congestion is imminent:

    ECN Mechanism

    1. Sender sets ECN-capable transport (ECT) bits in IP header
    2. Router marks packet (CE bit) instead of drop when congested
    3. Receiver echoes congestion indication to sender
    4. Sender reacts as if packet was dropped

    6.3 Other AQM Techniques

    CoDel (Controlled Delay)

    Uses sojourn time (time in queue) instead of queue length:

    • Targets persistent queues only
    • Adapts to changing link rates
    • No tuning parameters needed

    PIE (Proportional Integral controller Enhanced)

    Uses a control theory approach:

    • Estimates queuing delay
    • Adjusts drop probability based on deviation from target
    • Works well with shallow buffers

    FQ-CoDel (Fair Queuing + CoDel)

    Combines flow fairness with delay control:

    • Per-flow queuing
    • CoDel on each flow's queue
    • Prevents bufferbloat and flow starvation

    7. Challenges and Future Directions

    Current Challenges

    • High-speed networks: Traditional TCP performs poorly on high-BDP paths
    • Wireless networks: Packet loss ≠ congestion in mobile environments
    • Datacenter networks: Ultra-low latency requirements
    • Bufferbloat: Excessive buffering causes high latency
    • Fairness: Competing flows with different RTTs or algorithms

    Emerging Solutions

    QUIC Protocol

    Google's transport protocol with improved congestion control:

    • Uses TCP Cubic by default
    • Pluggable congestion control
    • Better loss recovery

    Machine Learning Approaches

    Using AI to optimize congestion control:

    • Reinforcement learning for parameter tuning
    • Neural networks for congestion prediction
    • Adaptive algorithms for diverse networks

    Information-centric Networking

    Alternative architectures that may reduce congestion:

    • Named Data Networking (NDN)
    • Content-centric approaches
    • In-network caching
