Software Engineering

    Unit Testing and Integration Testing

    1. Unit Testing

    1.1 Fundamentals of Unit Testing

    Unit testing involves testing individual components (units) of software in isolation to verify they function as intended.

    Key Characteristics

    • Isolated: Tests one unit at a time
    • Fast: Executes in milliseconds
    • Repeatable: Same results every time
    • Automated: Runs without manual intervention
    • Thorough: Covers all code paths

    Unit Testing Principles (FIRST)

    • Fast: Tests should run quickly
    • Isolated: No dependencies between tests
    • Repeatable: Consistent results in any environment
    • Self-validating: Clear pass/fail outcome
    • Timely: Written just before production code (TDD)

    1.2 Unit Testing Techniques

    Test Doubles

    Type Purpose When to Use
    Dummy Placeholder for unused parameters Required by signature but not used
    Stub Provides canned responses Testing simple indirect inputs
    Mock Pre-programmed expectations Behavior verification
    Fake Working lightweight implementation Replacing slow/difficult dependencies

    Example: Mocking in Python (unittest.mock)

                from unittest.mock import Mock
            
            # Create mock object
            payment_processor = Mock()
            
            # Set return value
            payment_processor.charge.return_value = True
            
            # Test function using mock
            def test_payment():
                result = process_payment(payment_processor, 100)
                assert result is True
                payment_processor.charge.assert_called_with(100)

    Popular Unit Testing Tools

    Language Framework Key Features

    Sample Unit Test Structure (JUnit 5)

                import org.junit.jupiter.api.*;
            import static org.junit.jupiter.api.Assertions.*;
            
            class CalculatorTest {
                
                Calculator calculator;
                
                @BeforeEach
                void setUp() {
                    calculator = new Calculator();
                }
                
                @Test
                @DisplayName("Test addition")
                void testAdd() {
                    assertEquals(5, calculator.add(2, 3));
                }
                
                @Test
                @DisplayName("Test division by zero")
                void testDivideByZero() {
                    assertThrows(ArithmeticException.class, () -> {
                        calculator.divide(10, 0);
                    });
                }
            }

    1.4 Advanced Unit Testing

    Test-Driven Development (TDD)

    1. Red: Write failing test for new functionality
    2. Green: Write minimal code to pass test
    3. Refactor: Improve code while keeping tests passing

    Parameterized Testing

    Run same test with different inputs:

                // JUnit 5 example
            @ParameterizedTest
            @ValueSource(ints = {1, 3, 5, -3, 15})
            void testIsOdd(int number) {
                assertTrue(Calculator.isOdd(number));
            }

    Mutation Testing

    Assesses test quality by introducing small bugs (mutations):

    • Tools: PITest (Java), Stryker (JavaScript)
    • Measures how many mutations are caught by tests
    • Identifies weak spots in test suite

    2. Integration Testing

    2.1 Fundamentals of Integration Testing

    Integration testing verifies that different modules or services work together as expected.

    Unit vs Integration Testing

    Aspect Unit Testing Integration Testing
    Scope Single component Interacting components
    Speed Milliseconds Seconds/minutes
    Dependencies Mocked/stubbed Real or test doubles
    Purpose Verify logic Verify interactions

    Integration Testing Challenges

    • Test environment setup complexity
    • External dependencies (databases, APIs)
    • Test data management
    • Test execution time
    • Debugging failures

    2.2 Integration Testing Strategies

    Approaches to Integration Testing

    Strategy Description Pros/Cons
    Big Bang Integrate all components at once Simple but hard to debug
    Top-Down Start with top-level, mock lower layers Early validation but incomplete
    Bottom-Up Start with low-level components Early bug detection but late UI testing
    Sandwich Combine top-down and bottom-up Balanced but complex

    Example: Microservices Integration

    Components: Order Service, Payment Service, Inventory Service
    Test Scenario:
    1. Order Service creates order
    2. Payment Service processes payment
    3. Inventory Service updates stock
    4. Verify all systems reflect consistent state

    2.3 Integration Testing Techniques

    Test Environment Management

    • Containerized dependencies: Docker for databases, message brokers
    • Service virtualization: Tools like WireMock, Mountebank
    • Test doubles: Fake services for external APIs
    • Test data management: Seed databases with known state

    Contract Testing

    Verifies interactions between services meet agreed contracts:

    • Tools: Pact, Spring Cloud Contract
    • Process:
      1. Consumer defines expected request/response
      2. Provider verifies it meets contract
      3. Tests run against real implementations

    Example: API Integration Test (Postman)

                // Test for User API
            pm.test("Create user returns 201", function() {
                pm.response.to.have.status(201);
            });
            
            pm.test("Response has user ID", function() {
                var jsonData = pm.response.json();
                pm.expect(jsonData.id).to.be.a('string');
            });
            
            pm.test("User exists in database", function() {
                pm.sendRequest({
                    url: pm.environment.get("db_service") + "/users/" + jsonData.id,
                    method: 'GET'
                }, function(err, response) {
                    pm.expect(response.code).to.equal(200);
                });
            });

    3. Testing Best Practices

    3.1 Effective Test Design

    Test Naming Conventions

    • Unit tests: MethodName_StateUnderTest_ExpectedBehavior
    • Integration tests: ComponentA_WithComponentB_ExpectedOutcome
    • Examples:
      - calculateTax_WithNegativeIncome_ThrowsException
      - OrderService_WithPaymentService_ProcessesCompleteOrder

    Test Structure (Arrange-Act-Assert)

                // Arrange - setup test conditions
            Calculator calc = new Calculator();
            int a = 5;
            int b = 3;
            
            // Act - execute the operation
            int result = calc.add(a, b);
            
            // Assert - verify outcomes
            assertEquals(8, result);

    3.2 Test Automation Strategy

    Testing Pyramid

    • Base (70%): Unit tests (fast, isolated)
    • Middle (20%): Integration tests (service interactions)
    • Top (10%): UI/E2E tests (whole system)

    CI/CD Integration

    • Run unit tests on every commit
    • Run integration tests on merge/pull request
    • Run extensive test suites nightly
    • Fail builds on test failures
    • Track test coverage metrics

    3.3 Managing Test Data

    Test Data Strategies

    • In-memory databases: H2, SQLite for fast tests
    • Test containers: Dockerized real databases
    • Data factories: Generate test objects
    • Golden datasets: Known good data snapshots

    Test Isolation Principles

    • Each test should setup its own data
    • Tests shouldn't depend on execution order
    • Clean up after tests (use teardown methods)
    • Consider transaction rollback for database tests

    4. Advanced Testing Topics

    4.1 Property-Based Testing

    Generate random inputs to test general properties rather than specific examples:

    Example: Hypothesis (Python)

                from hypothesis import given
            import hypothesis.strategies as st
            
            @given(st.integers(), st.integers())
            def test_add_commutative(a, b):
                assert add(a, b) == add(b, a)
            
            @given(st.lists(st.integers()))
            def test_reverse_twice_is_original(xs):
                assert reverse(reverse(xs)) == xs

    4.2 Mutation Testing

    Assess test quality by automatically introducing small bugs:

    PITest (Java) Example

                // Original code
            public boolean isPositive(int number) {
                return number > 0;
            }
            
            // Mutant generated (changed > to >=)
            public boolean isPositive(int number) {
                return number >= 0;
            }
            
            // Good test catches this mutation
            @Test
            void testIsPositive() {
                assertFalse(calculator.isPositive(0)); // kills mutant
            }

    4.3 Testing Legacy Code

    Strategies for Untested Code

    • Start with characterization tests
    • Use golden master technique
    • Extract methods to isolate testable units
    • Introduce seams for testability
    • Refactor incrementally with test coverage
