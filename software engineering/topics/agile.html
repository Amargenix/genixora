Software Engineering

    Agile Methodologies

    1. Agile Fundamentals

    1.1 What is Agile?

    Agile is an iterative approach to software development that emphasizes:

    • Customer collaboration over contract negotiation
    • Responding to change over following a plan
    • Working software over comprehensive documentation
    • Individuals and interactions over processes and tools

    12 Agile Principles

    1. Customer satisfaction through early delivery
    2. Welcome changing requirements
    3. Deliver working software frequently
    4. Business and developers work together
    5. Build projects around motivated individuals
    6. Face-to-face conversation is best
    7. Working software is primary measure
    8. Sustainable development pace
    9. Continuous attention to technical excellence
    10. Simplicity is essential
    11. Self-organizing teams
    12. Regular reflection and adjustment

    1.2 Agile vs Waterfall

    Characteristic Waterfall Agile
    Approach Linear and sequential Iterative and incremental
    Flexibility Difficult to change Embraces change
    Delivery Single final delivery Frequent small deliveries
    Customer Involvement Mainly at start and end Continuous collaboration
    Documentation Extensive upfront Just enough, just in time

    2. Agile Frameworks

    2.1 Scrum

    Scrum Overview

    An iterative framework with fixed-length iterations (sprints) typically lasting 1-4 weeks.

    Scrum Events

    Event Duration Purpose
    Sprint Planning 2-4 hours/week Plan work for upcoming sprint
    Daily Scrum 15 min/day Synchronize team activities
    Sprint Review 1-2 hours/week Inspect increment and adapt
    Sprint Retrospective 1-2 hours/week Improve team processes

    Scrum Artifacts

    • Product Backlog: Prioritized list of features
    • Sprint Backlog: Items selected for current sprint
    • Increment: Working product at sprint end

    2.2 Kanban

    Kanban Overview

    A visual workflow management method focusing on continuous delivery without iterations.

    Kanban Practices

    1. Visualize the workflow
    2. Limit work in progress (WIP)
    3. Manage flow
    4. Make policies explicit
    5. Implement feedback loops
    6. Improve collaboratively

    Scrum vs Kanban

    Aspect Scrum Kanban
    Iterations Fixed-length sprints Continuous flow
    Commitment Sprint backlog commitment No required commitment
    Roles Product Owner, Scrum Master No prescribed roles
    Change Discouraged mid-sprint Allowed anytime

    2.3 Other Agile Frameworks

    Extreme Programming (XP)

    • Values: Communication, simplicity, feedback, courage
    • Practices: Pair programming, test-driven development, continuous integration
    • Best for: Teams needing technical discipline

    Lean Software Development

    • Principles: Eliminate waste, amplify learning, decide late, deliver fast
    • Focus: Value stream optimization
    • Best for: Process improvement initiatives

    Scaled Agile Framework (SAFe)

    • Structure: Portfolio, program, team levels
    • Events: PI Planning, System Demo
    • Best for: Large enterprises (100+ people)

    3. Agile Implementation

    3.1 Agile Roles

    Scrum Roles

    • Product Owner: Represents stakeholders, manages backlog
    • Scrum Master: Facilitates process, removes impediments
    • Development Team: Cross-functional, self-organizing

    Extended Agile Roles

    • Stakeholders: Customers, users, executives
    • Agile Coach: Helps teams adopt Agile
    • Architect: Technical guidance (in some frameworks)

    3.2 Agile Artifacts

    Product Backlog

    • Prioritized list of all desired work
    • Contains user stories, bugs, technical work
    • Continuously refined (grooming)
    • Owned by Product Owner

    Example: User Story Format

    As a [role]
    I want [feature]
    So that [benefit]

    Acceptance Criteria:
    - Must show confirmation after submission
    - Should validate email format
    - Must log failed attempts

    3.3 Agile Estimation

    Story Points

    • Relative measure of complexity/effort
    • Fibonacci sequence (1, 2, 3, 5, 8, 13)
    • Based on team velocity (points per sprint)
    • Accounts for risk and uncertainty

    Planning Poker

    1. Product Owner explains story
    2. Team discusses briefly
    3. Each member privately selects estimate
    4. All reveal estimates simultaneously
    5. Discuss differences and re-estimate

    4. Agile Best Practices

    4.1 Effective Agile Teams

    Team Characteristics

    • Cross-functional: All skills needed to deliver
    • Co-located: Prefer physical proximity
    • Empowered: Authority to make decisions
    • Stable: Minimal member changes
    • Optimal size: 5-9 members

    Daily Standup Tips

    1. Stand to keep meetings short
    2. Answer three questions:
      - What did I do yesterday?
      - What will I do today?
      - What impediments do I have?
    3. Focus on progress toward sprint goal
    4. Address detailed discussions offline

    4.2 Agile Technical Practices

    Continuous Integration

    • Developers merge code frequently (multiple times/day)
    • Automated builds and tests run on each check-in
    • Fast feedback on integration problems

    Test-Driven Development (TDD)

    1. Write failing test for small feature
    2. Write minimal code to pass test
    3. Refactor code while keeping tests passing

    Pair Programming

    • Two developers work together at one workstation
    • "Driver" writes code, "navigator" reviews
    • Roles switch frequently
    • Improves quality and knowledge sharing

    4.3 Common Pitfalls

    Agile Anti-Patterns

    • ScrumBut: "We do Scrum, but..." (missing key elements)
    • Water-Scrum-Fall: Waterfall disguised with sprints
    • Feature Factory: Focus on output over outcomes
    • Zombie Scrum: Going through motions without value

    Overcoming Challenges

    • Distributed Teams: Over-communicate, use collaboration tools
    • Fixed-Price Contracts: Time-boxed iterations with variable scope
    • Enterprise Adoption: Start small, demonstrate success
