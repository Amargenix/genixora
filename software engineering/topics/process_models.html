Software Engineering

    Software Process Models

    1. Introduction to Software Process Models

    A software process model is a simplified representation of a software process, presented from a specific perspective. These models describe the sequence of activities in the process and the flow between these activities.

    Why Process Models Matter

    • Provide structure to development activities
    • Help estimate costs and schedules
    • Facilitate communication among stakeholders
    • Enable process improvement
    • Reduce project risks

    Common Challenges

    • No one-size-fits-all model
    • Rigid models may not adapt to change
    • Overhead of following process
    • Difficulty in selecting appropriate model

    Software Process Model Spectrum

    1

    Waterfall Model

    The classic linear sequential approach to software development

    1

    Requirements Analysis

    All requirements are gathered and documented in detail before any development begins.

    2

    System Design

    Software architecture and system design are created based on requirements.

    3

    Implementation

    Actual coding of the software takes place in this phase.

    4

    Testing

    The developed software is tested against the requirements.

    5

    Deployment

    The system is deployed in the customer environment.

    6

    Maintenance

    Bug fixes and updates are performed based on user feedback.

    Advantages

    • Simple and easy to understand
    • Clear milestones and deliverables
    • Good for stable requirements
    • Well-documented process
    • Easy to manage due to rigidity

    Disadvantages

    • Inflexible to changing requirements
    • Working software late in the cycle
    • High risk and uncertainty
    • Difficult to measure progress
    • Poor model for complex projects

    Case Study: Waterfall in Aerospace

    The NASA Space Shuttle flight software was developed using a waterfall approach due to:

    • Extremely stable requirements
    • Critical need for reliability
    • Rigorous documentation needs
    • Highly predictable development process

    The result was one of the most reliable software systems ever built, with an estimated error rate of less than 1 defect per 400,000 lines of code.

    2

    V-Model (Verification and Validation Model)

    Extension of waterfall model with emphasis on testing throughout the lifecycle

    V-Model Diagram

    Requirements Analysis

    User requirements gathering

    Acceptance Testing

    Validates against user requirements

    System Design

    High-level architecture

    System Testing

    Validates complete system

    Module Design

    Detailed component design

    Integration Testing

    Tests module interactions

    Implementation

    Coding of components

    Unit Testing

    Tests individual components

    Advantages

    • Emphasizes testing throughout lifecycle
    • Defects detected early
    • Clear validation for each development stage
    • Good for high reliability systems
    • Works well for small to medium projects

    Disadvantages

    • Inflexible to changes
    • High cost of changes
    • Working software late in cycle
    • Not suitable for complex projects
    • Requires detailed planning

    Quick Quiz

    Q: What is the main difference between the V-Model and the Waterfall Model?

    A: The V-Model explicitly pairs each development phase with a corresponding testing phase, emphasizing verification and validation throughout the lifecycle, while the Waterfall Model treats testing as a separate phase at the end.
