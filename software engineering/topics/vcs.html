Software Engineering

    Version Control Systems (Git & SVN)

    1. Version Control Fundamentals

    1.1 What is Version Control?

    A Version Control System (VCS) tracks changes to files over time so you can:

    • Recall specific versions later
    • Collaborate with multiple developers
    • Revert files or entire projects to previous states
    • Compare changes over time
    • Identify who made changes and why

    Key Benefits

    • Collaboration: Multiple people work simultaneously
    • Backup: Complete history of changes
    • Experimentation: Safe environment to try ideas
    • Accountability: Track who changed what
    • Reproducibility: Recreate any project state

    1.2 Types of Version Control

    Centralized vs Distributed

    Aspect Centralized (SVN) Distributed (Git)
    Repository Single central server Every clone is full repository
    Workflow Checkout-edit-commit Clone-edit-stage-commit-push
    Network Needed for most operations Only needed to share changes
    Branching Heavyweight (directory copies) Lightweight (instant branches)
    Speed Slower for remote operations Most operations are local

    2. Git - Distributed Version Control

    2.1 Git Architecture

    Three Main Areas

    1. Working Directory: Actual files you edit
    2. Staging Area (Index): Prepares changes for commit
    3. Git Directory (.git): Where Git stores metadata and object database

    Key Concepts

    • Snapshot: Git records complete file states (not diffs)
    • SHA-1 Hash: 40-character checksum identifies all objects
    • Immutable Objects: Once created, objects never change

    2.2 Basic Git Workflow

    Daily Commands

    1. git init - Create new repository
    2. git clone [url] - Copy existing repository
    3. git status - View changed files
    4. git add [file] - Stage changes
    5. git commit -m "message" - Record changes
    6. git push - Send changes to remote
    7. git pull - Fetch and merge changes

    Example: First-Time Setup

    # Configure user information
    git config --global user.name "John Doe"
    git config --global user.email "john@example.com"

    # Create new project
    mkdir my-project
    cd my-project
    git init

    # Add and commit first file
    echo "# My Project" > README.md
    git add README.md
    git commit -m "Initial commit"

    2.3 Branching and Merging

    Branch Commands

    # List all branches
    git branch

    # Create new branch
    git branch feature-login

    # Switch to branch
    git checkout feature-login
    # Or (newer syntax)
    git switch feature-login

    # Create and switch in one command
    git checkout -b feature-login

    # Merge branch into current branch
    git merge feature-login

    # Delete branch
    git branch -d feature-login

    Merge Strategies

    • Fast-forward: When no divergent changes exist
    • 3-way merge: Creates new merge commit when branches diverge
    • Rebase: Moves branch commits to tip of target branch

    2.4 Remote Repositories

    Working with Remotes

    # List configured remotes
    git remote -v

    # Add new remote
    git remote add origin https://github.com/user/repo.git

    # Push branch to remote
    git push -u origin main

    # Fetch changes from remote
    git fetch

    # Pull changes (fetch + merge)
    git pull

    # Clone existing repository
    git clone https://github.com/user/repo.git

    Common Hosting Services

    • GitHub
    • GitLab
    • Bitbucket
    • Azure DevOps

    2.5 Advanced Git

    Undoing Changes

    # Unstage file (keep changes)
    git reset HEAD file.txt

    # Discard local changes
    git checkout -- file.txt

    # Amend last commit
    git commit --amend

    # Revert a commit (creates new undo commit)
    git revert abc123

    # Reset to previous commit (caution!)
    git reset --hard abc123

    Rebasing

    # Rebase current branch onto main
    git rebase main

    # Interactive rebase (last 3 commits)
    git rebase -i HEAD~3

    # Common interactive commands:
    # p = pick (use commit)
    # r = reword (edit commit message)
    # e = edit (amend commit)
    # s = squash (combine with previous)

    3. SVN - Centralized Version Control

    3.1 SVN Architecture

    Repository Structure

    • Central Repository: Single source of truth
    • Working Copy: Local snapshot of files
    • Revisions: Numbered snapshots of entire repository

    Key Differences from Git

    • No local repository - only working copy
    • Revision numbers apply to entire repository
    • Branching implemented via directory copies
    • Requires network connection for most operations

    3.2 Basic SVN Workflow

    Daily Commands

    # Checkout repository
    svn checkout http://svn.example.com/repo/trunk

    # Update working copy
    svn update

    # Check status of files
    svn status

    # Add new file to version control
    svn add file.txt

    # Commit changes
    svn commit -m "Added new feature"

    # View commit history
    svn log

    3.3 Branching in SVN

    Branch/Tag Operations

    # Create branch (copy trunk to branches)
    svn copy http://svn.example.com/repo/trunk \
    http://svn.example.com/repo/branches/feature-x \
    -m "Creating feature-x branch"

    # Switch working copy to branch
    svn switch http://svn.example.com/repo/branches/feature-x

    # Merge changes from trunk to branch
    svn merge http://svn.example.com/repo/trunk

    # Create tag (similar to branch)
    svn copy http://svn.example.com/repo/trunk \
    http://svn.example.com/repo/tags/v1.0 \
    -m "Tagging version 1.0"

    Important Notes

    • SVN branches are just directory copies (cheap but not as flexible as Git)
    • Merging requires careful tracking of revision numbers
    • Always update before committing to avoid conflicts

    3.4 Advanced SVN

    Useful Commands

    # View differences
    svn diff
    svn diff -r 100:200 file.txt

    # Revert local changes
    svn revert file.txt

    # Resolve conflicts
    svn resolve --accept=mine-full file.txt

    # View file history
    svn blame file.txt

    # Export clean copy (no .svn folders)
    svn export http://svn.example.com/repo/trunk

    4. Git vs SVN Comparison

    4.1 When to Use Each

    Consideration Git SVN
    Project Size Excellent for large projects Better for smaller, centralized teams
    Internet Access Works offline Requires server connection
    Branching Lightweight, frequent branching Heavyweight, less frequent
    Learning Curve Steeper initially Simpler basic workflow
    Binary Files Handles them but not ideally Better with large binaries

    4.2 Migration Between Systems

    SVN to Git Migration

    # Install git-svn
    sudo apt-get install git-svn

    # Clone SVN repo with history
    git svn clone http://svn.example.com/repo \
    --authors-file=users.txt \
    --stdlayout my-git-repo

    # Clean up branches and tags
    cd my-git-repo
    git remote add origin git@github.com:user/repo.git
    git push -u origin --all
    git push -u origin --tags

    5. Version Control Best Practices

    5.1 General Guidelines

    Commit Practices

    • Commit often, perfect later, publish once
    • Each commit should represent one logical change
    • Write meaningful commit messages
    • Don't commit generated files or secrets

    Branching Strategies

    • Git Flow: Feature, develop, release, hotfix branches
    • GitHub Flow: Simple main branch + feature branches
    • Trunk-Based Development: Small frequent commits to main

    5.2 Collaboration Practices

    Team Workflows

    • Agree on branching strategy upfront
    • Use pull requests for code review
    • Rebase feature branches before merging
    • Resolve merge conflicts carefully

    .gitignore Tips

    • Ignore OS-specific files (.DS_Store, thumbs.db)
    • Ignore IDE/config files (.idea, .vscode)
    • Ignore dependencies (node_modules, .venv)
    • Ignore build artifacts (bin, obj, *.exe)
