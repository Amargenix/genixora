    Unified Modeling Language (UML) Diagrams

    1. UML Fundamentals

    1.1 What is UML?

    The Unified Modeling Language (UML) is a standardized modeling language for:

    • Visualizing system designs
    • Specifying system architecture
    • Documenting software components
    • Communicating design decisions

    UML 2.x Diagram Categories

    Category Diagram Types Purpose
    Structural Class, Object, Component, Deployment, Composite Structure, Package, Profile System organization and structure
    Behavioral Use Case, Activity, State Machine, Sequence, Communication, Timing, Interaction Overview System behavior and interactions

    1.2 UML Notation Basics

    Common UML Elements

    • Actors: External entities interacting with system
    • Classes: Blueprint for objects (name, attributes, methods)
    • Objects: Instances of classes
    • Interfaces: Contract of operations
    • Relationships: Associations, dependencies, generalizations

    Relationship Types

    Relationship Notation Description
    Association ────── Structural relationship between classes
    Dependency ─────▷ One element uses/needs another
    Generalization ──────▷ Inheritance (parent-child relationship)
    Realization ──────▷ Class implements an interface
    Aggregation ──────◇ "Has-a" relationship (weak containment)
    Composition ──────◆ Strong "whole-part" relationship

    2. Structural Diagrams

    Class Diagram Elements

    • Class: Rectangle with 3 compartments (name, attributes, methods)
    • Visibility: + (public), - (private), # (protected), ~ (package)
    • Multiplicity: 1 (exactly one), 0..1 (zero or one), * (many), 1..* (one or more)

    Example: E-Commerce System

    Classes: Customer, Order, Product, Payment
    Relationships:
    - Customer places Order (association)
    - Order contains Products (composition)
    - Payment depends on Order (dependency)

    2.2 Object Diagrams

    Class vs Object Diagrams

    Aspect Class Diagram Object Diagram
    Purpose Shows classes and relationships Shows objects at a specific time
    Elements Classes, interfaces Objects (instances)
    Usage Design phase Runtime snapshot

    Example: Library System

    Objects:
    - customer1:Customer (name="John", id=101)
    - book1:Book (title="UML Guide", isbn="12345")
    Links: customer1--borrows-->book1

    2.3 Component Diagrams

    Component Diagram Elements

    • Component: Rectangular box with component icon
    • Interface: Lollipop (provided), socket (required)
    • Port: Interaction point between component and environment

    Example: Online Banking System

    Components:
    - User Interface
    - Account Management
    - Transaction Processing
    - Database Access
    Connections: Required/provided interfaces between components

    2.4 Deployment Diagrams

    Deployment Diagram Elements

    • Node: Hardware/software execution environment
    • Artifact: Physical representation of software
    • Communication Path: Connection between nodes

    Example: Web Application Deployment

    Nodes:
    - Client Browser
    - Web Server (Apache)
    - Application Server (Tomcat)
    - Database Server (MySQL)
    Artifacts: WebApp.war, DatabaseSchema.sql

    3. Behavioral Diagrams

    3.1 Use Case Diagrams

    Use Case Elements

    • Actor: Stick figure (system user or external system)
    • Use Case: Oval (system functionality)
    • Relationships:
      - Association (actor to use case)
      - Include (mandatory functionality)
      - Extend (optional functionality)
      - Generalization (specialization)

    Example: ATM System

    Actors: Customer, Bank, Maintenance Technician
    Use Cases:
    - Withdraw Cash
    - Check Balance
    - Transfer Funds
    - Service Machine (extends Maintenance)

    3.2 Sequence Diagrams

    Sequence Diagram Elements

    • Lifeline: Vertical dashed line (object's existence)
    • Activation Bar: Rectangle on lifeline (method execution)
    • Messages: Arrows between lifelines (synchronous, asynchronous, return)
    • Combined Fragments: alt, opt, loop, par (for logic)

    Example: User Login

    Objects: UserInterface, LoginController, UserDB
    Messages:
    1. enterCredentials()
    2. validateUser()
    3. [valid] getUserDetails()
    4. return success

    3.3 Activity Diagrams

    Activity Diagram Elements

    • Initial Node: Filled circle (start point)
    • Activity: Rounded rectangle (action)
    • Decision Node: Diamond (branching)
    • Fork/Join: Horizontal bar (parallel flows)
    • Final Node: Bullseye (end point)

    Example: Order Processing

    Flow:
    Start → Receive Order → [In Stock] Process Payment → Ship Order → End
    [Out of Stock] Notify Customer → End

    3.4 State Machine Diagrams

    State Diagram Elements

    • State: Rounded rectangle (system condition)
    • Transition: Arrow (state change)
    • Event: Trigger for transition
    • Action: Activity during transition
    • Initial/Final States: Start/end points

    Example: Elevator Control

    States: Idle, Moving Up, Moving Down, Stopped
    Transitions:
    - floorRequested → Moving Up/Down
    - reachedDestination → Stopped
    - doorOpened → Idle

    Visual Paradigm Desktop/Online Agile support, team collaboration
    PlantUML Text-based Version control friendly

    4.2 UML Best Practices

    Effective UML Modeling

    • Start with high-level diagrams (use cases, components)
    • Use consistent naming conventions
    • Keep diagrams focused (7±2 elements per diagram)
    • Use notes/comments for clarification
    • Validate diagrams with stakeholders
    • Maintain traceability between diagrams

    Common Mistakes to Avoid

    • Overly complex diagrams
    • Inconsistent abstraction levels
    • Missing key relationships
    • Not updating diagrams with code changes
    • Creating diagrams without purpose

    5. Advanced UML Concepts

    5.1 Interaction Diagrams

    Sequence vs Communication vs Timing

    Diagram Focus Best For
    Sequence Message ordering over time Detailed method calls
    Communication Object relationships Architectural overview
    Timing State changes over time Real-time systems

    5.2 Composite Structure

    When to Use Composite Structure

    • Modeling complex component internals
    • Showing parts and connectors
    • Representing runtime architectures
    • Modeling hardware/software interfaces

    5.3 Profile Diagrams

    Extending UML with Stereotypes

    • Create domain-specific UML extensions
    • Define custom stereotypes, tagged values
    • Apply to existing UML elements
    • Example: «Service», «Entity», «Controller»
