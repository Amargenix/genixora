Software Engineering

    Software Design Principles

    1. Introduction to Design Principles

    Software design principles are fundamental guidelines that help developers create maintainable, scalable, and robust systems.

    Why Design Principles Matter

    • Maintainability: Easier to modify and extend
    • Reusability: Components can be used in multiple contexts
    • Testability: Easier to verify correctness
    • Collaboration: Common understanding among team members

    Principles vs. Patterns

    Design Principles Design Patterns
    Fundamental guidelines Specific implementations
    Abstract concepts Concrete solutions
    SOLID, DRY, KISS Factory, Observer, Singleton

    2. SOLID Principles

    The SOLID principles are five object-oriented design principles that make software designs more understandable, flexible, and maintainable.

    S - Single Responsibility Principle (SRP)

    "A class should have only one reason to change"

    Example:

    // Good: Separate report generation from data storage class User { constructor(name) { this.name = name; } getName() { return this.name; } } class UserReport { generate(user) { console.log(`User: ${user.getName()}`); } }

    Violation:

    // Bad: User class handles both data and reporting class User { constructor(name) { this.name = name; } getName() { return this.name; } generateReport() { console.log(`User: ${this.name}`); } }

    O - Open/Closed Principle (OCP)

    "Software entities should be open for extension but closed for modification"

    Example:

    // Good: Extend through interfaces interface Shape { area(): number; } class Circle implements Shape { constructor(private radius: number) {} area() { return Math.PI * this.radius ** 2; } } class Square implements Shape { constructor(private side: number) {} area() { return this.side ** 2; } }

    L - Liskov Substitution Principle (LSP)

    "Subtypes must be substitutable for their base types"

    Example:

    // Good: Rectangle and Square both behave as expected class Rectangle { constructor(protected width, protected height) {} setWidth(w) { this.width = w; } setHeight(h) { this.height = h; } area() { return this.width * this.height; } } class Square extends Rectangle { setSize(size) { this.width = this.height = size; } setWidth(w) { this.setSize(w); } setHeight(h) { this.setSize(h); } }

    I - Interface Segregation Principle (ISP)

    "Clients shouldn't be forced to depend on interfaces they don't use"

    Example:

    // Good: Separate interfaces interface Printer { print(document: Document); } interface Scanner { scan(document: Document); } class Photocopier implements Printer, Scanner { print(document: Document) { /* ... */ } scan(document: Document) { /* ... */ } }

    D - Dependency Inversion Principle (DIP)

    "Depend on abstractions, not concretions"

    Example:

    // Good: Depend on Logger interface interface Logger { log(message: string); } class FileLogger implements Logger { log(message: string) { /* Write to file */ } } class App { constructor(private logger: Logger) {} run() { this.logger.log("App started"); } }

    3. Other Fundamental Principles

    DRY (Don't Repeat Yourself)

    "Every piece of knowledge must have a single, unambiguous representation"

    Example:

    // Bad: Duplicate validation logic function processUser(user) { if (!user.name) throw new Error("Name required"); // ... } function createUser(user) { if (!user.name) throw new Error("Name required"); // ... } // Good: Centralized validation function validateUser(user) { if (!user.name) throw new Error("Name required"); } function processUser(user) { validateUser(user); // ... }

    KISS (Keep It Simple, Stupid)

    "Simplicity should be a key goal in design"

    Example:

    // Bad: Over-engineered solution function getDaySuffix(day) { const suffixes = { 1: 'st', 2: 'nd', 3: 'rd', 21: 'st', 22: 'nd', 23: 'rd', 31: 'st' }; return suffixes[day] || 'th'; } // Good: Simple solution function getDaySuffix(day) { if (day % 100 >= 11 && day % 100 <= 13) return 'th'; switch (day % 10) { case 1: return 'st'; case 2: return 'nd'; case 3: return 'rd'; default: return 'th'; } }

    4. Composition Over Inheritance

    Favor object composition over class inheritance for more flexible and maintainable code.

    Inheritance Approach

    class Bird { fly() { console.log("Flying"); } } class Penguin extends Bird { fly() { throw new Error("Can't fly"); } }

    Problems:

    • Violates LSP (Penguin isn't substitutable for Bird)
    • Creates fragile base class problem

    Composition Approach

    class FlyingBehavior { fly() { console.log("Flying"); } } class Bird { constructor(flyingBehavior) { this.flyingBehavior = flyingBehavior; } } const flyingBird = new Bird(new FlyingBehavior()); const penguin = new Bird(null);

    Benefits:

    • More flexible
    • Easier to modify behavior
    • No inheritance hierarchy issues

    5. Law of Demeter (Principle of Least Knowledge)

    "Only talk to your immediate friends" - reduces coupling between components.

    Rules of Demeter

    1. An object can call its own methods
    2. An object can call methods of its properties
    3. An object can call methods of objects it creates
    4. An object can call methods of objects passed as parameters

    Violation:

    // Bad: Traversing multiple object relationships function printOrderTotal(order) { const customer = order.getCustomer(); const address = customer.getAddress(); const country = address.getCountry(); console.log(country.getName()); }

    Proper Implementation:

    // Good: Delegate to immediate friends function printOrderTotal(order) { console.log(order.getCustomerCountryName()); } // In Order class: getCustomerCountryName() { return this.customer.getCountryName(); } // In Customer class: getCountryName() { return this.address.getCountryName(); }

    6. YAGNI (You Aren't Gonna Need It)

    "Don't implement functionality until it's actually necessary"

    Why YAGNI Matters

    • Reduces unnecessary complexity
    • Saves development time
    • Makes code easier to modify
    • Prevents over-engineering

    Example of Over-Engineering:

    // Bad: Implementing flexible logging system when simple console.log would suffice class Logger { private strategy: LogStrategy; setStrategy(strategy: LogStrategy) { this.strategy = strategy; } log(message: string) { this.strategy.log(message); } } interface LogStrategy { log(message: string): void; } // When all you needed was: console.log("Something happened");

    8. High Cohesion

    "Related functionality should be kept together"

    Characteristics of Cohesive Modules

    • All elements work toward a single purpose
    • Minimal interaction with other modules
    • Easy to understand and maintain

    Example:

    // Good: All authentication-related code together class AuthService { login(username, password) { /* ... */ } logout() { /* ... */ } refreshToken() { /* ... */ } validateToken() { /* ... */ } } // Bad: Auth methods scattered across different classes class UserService { login() { /* ... */ } } class SecurityService { validateToken() { /* ... */ } } class AppController { logout() { /* ... */ } }

    9. Loose Coupling

    "Components should have minimal knowledge of other components"

    Coupling Reduction Techniques

    • Use interfaces instead of concrete implementations
    • Apply Dependency Injection
    • Follow Law of Demeter
    • Use event-driven architecture

    Tight Coupling

    class MySQLDatabase { save(data) { /* ... */ } } class UserService { constructor() { this.db = new MySQLDatabase(); } createUser(user) { this.db.save(user); } }

    Loose Coupling

    interface Database { save(data): void; } class UserService { constructor(private db: Database) {} createUser(user) { this.db.save(user); } }

    10. Principle Summary & Cheat Sheet

    Design Principle Acronyms

    Acronym Principle Key Benefit
    SOLID Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion Maintainable OO design
    DRY Don't Repeat Yourself Reduces duplication
    KISS Keep It Simple, Stupid Prevents over-engineering
    YAGNI You Aren't Gonna Need It Avoids unnecessary features

    When to Apply Which Principle

    Situation Applicable Principles
    Class doing too much SRP, Separation of Concerns
    Hard to add features OCP, Composition over Inheritance
    Changing one module breaks others Low Coupling, DIP
    Duplicate code DRY, High Cohesion
