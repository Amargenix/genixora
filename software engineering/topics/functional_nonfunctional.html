Software Engineering

    Functional and Non-Functional Requirements in Software Engineering

    1. Introduction to Software Requirements

    Software requirements are the foundation of any successful software project. They define what the system should do and how it should perform.

    Functional Requirements

    Define what the system should do - the features and functionalities.

    Non-Functional Requirements

    Define how well the system performs its functions - the quality attributes.

    Example: Online Shopping System

    Functional: "The system shall allow users to add items to a shopping cart."

    Non-Functional: "The system shall respond to user actions within 2 seconds during peak hours."

    2. Functional Requirements

    Functional requirements specify the behaviors and operations of a system.

    Characteristics of Good Functional Requirements:

    • ✓ Complete and unambiguous
    • ✓ Consistent and non-conflicting
    • ✓ Verifiable and testable
    • ✓ Traceable to business needs
    • ✓ Prioritized for implementation
    1

    Business Function Requirements

    High-level requirements that map directly to business objectives.

    /** * Business Function Requirement Example: * The system shall process customer orders * and generate invoices automatically. */
    2

    User Function Requirements

    Specific actions users can perform with the system.

    User Role Functionality
    Customer Browse product catalog, place orders
    Admin Manage inventory, view reports

    3. Non-Functional Requirements

    Non-functional requirements define the quality attributes and constraints of the system.

    Categories of Non-Functional Requirements:

    Category Description Examples
    Performance Speed, efficiency, resource usage "System must handle 1000 concurrent users"
    Security Protection against threats "All passwords must be encrypted"
    Usability User experience and interface "90% of users should complete tasks without help"
    Reliability Availability and fault tolerance "99.9% uptime guarantee"
    Scalability Ability to handle growth "System should scale to 10x current load"

    Well-Structured Non-Functional Requirement:

    "The search functionality shall return results for simple queries within 1 second for 95% of the queries when the database contains up to 1 million records."

    Why it's good: Specific, measurable, and includes conditions.

    Poorly-Structured Non-Functional Requirement:

    "The system should be fast."

    Why it's bad: Vague, not measurable, no clear acceptance criteria.

    4. Requirements Gathering Techniques

    1

    Interviews

    One-on-one discussions with stakeholders to understand needs.

    • Pros: Detailed information, personal interaction
    • Cons: Time-consuming, may be biased
    2

    Surveys and Questionnaires

    Structured questions distributed to many stakeholders.

    • Pros: Scalable, quantitative data
    • Cons: Limited depth, low response rates
    3

    Workshops

    Group sessions with multiple stakeholders.

    • Pros: Collaborative, immediate feedback
    • Cons: Dominant personalities may skew results
    4

    Observation

    Watching users perform their current tasks.

    • Pros: Reveals actual workflows
    • Cons: Hawthorne effect (people change behavior when observed)

    5. Documenting Requirements

    Common Documentation Formats:

    Format Best For Example Tools
    Software Requirements Specification (SRS) Formal, comprehensive documentation Word, Confluence
    User Stories Agile development Jira, Trello
    Use Cases Detailed interaction flows UML tools, Lucidchart
    Requirements Traceability Matrix Tracking requirements through lifecycle Excel, DOORS

    SRS Template Structure:

    1. Introduction (Purpose, Scope, Definitions)
    2. Overall Description (Product Perspective, Functions)
    3. System Features (Functional Requirements)
    4. External Interface Requirements
    5. Non-Functional Requirements
    6. Other Requirements (Legal, Compliance)
    /** * User Story Example: * As a [role], I want [feature] so that [benefit]. * * As a customer, I want to filter products by price range * so that I can find items within my budget. */

    6. Requirements Validation

    Ensuring requirements accurately capture stakeholder needs and are properly specified.

    Validation Techniques:

    Reviews

    • Structured walkthroughs
    • Checklist-based evaluation
    • Peer reviews

    Prototyping

    • Throw-away prototypes
    • Evolutionary prototypes
    • Wireframes and mockups

    Requirements Validation Checklist:

    • ✓ Are requirements complete?
    • ✓ Are requirements consistent?
    • ✓ Are requirements realistic?
    • ✓ Are requirements verifiable?
    • ✓ Are requirements prioritized?
    • ✓ Are requirements traceable?

    Quick Quiz

    Q: What is the main difference between validation and verification in requirements engineering?

    A: Validation ensures we're building the right product (meeting user needs), while verification ensures we're building the product right (meeting specifications).

    7. Requirements Management

    The process of documenting, analyzing, tracing, prioritizing, and controlling changes to requirements.

    Key Aspects of Requirements Management:

    1

    Version Control

    Tracking changes to requirements over time

    2

    Traceability

    Linking requirements to design, implementation, and testing

    3

    Change Control

    Formal process for evaluating and implementing changes

    Requirement ID Description Source Priority Status
    FR-101 User login functionality Stakeholder interview High Implemented
    NFR-205 Response time < 2s Business requirement Medium In progress

    8. Case Studies

    Case Study 1: Banking System

    Challenge: Complex regulatory requirements and high security needs.

    Solution: Detailed functional requirements for transaction processing combined with rigorous non-functional requirements for security and compliance.

    Key Requirements:

    • FR: System shall process wire transfers between accounts
    • NFR: All financial transactions shall be encrypted with AES-256
    • NFR: System shall maintain audit logs of all transactions

    Case Study 2: E-Commerce Platform

    Challenge: Need to handle seasonal traffic spikes while maintaining performance.

    Solution: Comprehensive performance and scalability requirements with cloud-based architecture.

    Key Requirements:

    • FR: System shall recommend products based on browsing history
    • NFR: System shall handle 10,000 concurrent users during peak
    • NFR: Product pages shall load within 1.5 seconds

    10. Best Practices

    For Functional Requirements:

    • ✓ Use consistent templates and formats
    • ✓ Employ "shall" statements for mandatory requirements
    • ✓ Include error handling scenarios
    • ✓ Organize by feature or user role
    • ✓ Use diagrams to complement text

    For Non-Functional Requirements:

    • ✓ Make them measurable and testable
    • ✓ Specify conditions and constraints
    • ✓ Prioritize based on business impact
    • ✓ Consider trade-offs between competing NFRs
    • ✓ Document assumptions

    SMART Requirements:

    Specific, Measurable, Achievable, Relevant, Time-bound

    Instead of: "The system should be fast"

    Use: "The search results page shall load within 1 second for 95% of queries when the database contains up to 1 million records"

    11. Conclusion

    Understanding and properly documenting both functional and non-functional requirements is critical to software project success. While functional requirements define what the system should do, non-functional requirements determine how well it performs these functions.

    Key Takeaways:

    1. Functional requirements describe system behaviors and features
    2. Non-functional requirements define quality attributes
    3. Both types should be specific, measurable, and testable
    4. Proper requirements management reduces project risks
    5. Tools and templates can streamline the requirements process
