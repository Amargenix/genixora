    Agile Testing

    Fundamentals of Agile Testing

    Definition

    Agile testing is a software testing practice that follows the principles of Agile development, where testing occurs continuously throughout the development lifecycle rather than as a separate phase.

    Key Principles

    • Testing is continuous, not a phase
    • Whole team approach to quality
    • Fast feedback loops
    • Tests as executable specifications
    • Automation where valuable

    Traditional vs. Agile Testing

    Traditional Agile
    Phase at project end Continuous activity
    Separate test team Whole team responsibility
    Documentation heavy Executable specifications

    Agile Testing Quadrants

    Q1: Technology Facing (Supporting Team)

    • Unit tests
    • Component tests
    • Test-driven development (TDD)
    • Automated regression tests

    Q2: Business Facing (Supporting Team)

    • Functional tests
    • Story tests
    • Examples and scenarios
    • Behavior-driven development (BDD)

    Q3: Business Facing (Critiquing Product)

    • Exploratory testing
    • Scenario testing
    • Usability testing
    • User acceptance testing

    Q4: Technology Facing (Critiquing Product)

    • Performance testing
    • Load testing
    • Security testing
    • Compliance testing

    Note: Quadrants are not sequential phases but categories of testing activities that occur throughout the Agile lifecycle.

    Testing in Scrum Framework

    Testing in Sprint Planning

    • Testers participate in backlog refinement
    • Help define acceptance criteria
    • Estimate testing effort for stories
    • Identify test dependencies

    Daily Testing Activities

    • Automated regression testing
    • Exploratory testing sessions
    • Collaboration with developers
    • Continuous integration testing

    Sprint Review Testing

    • Demo of tested functionality
    • Show test coverage and results
    • Gather feedback for new tests
    • Validate business value delivered

    Retrospective Focus

    • Test automation effectiveness
    • Testing bottlenecks
    • Quality metrics review
    • Process improvements

    Agile Testing Techniques

    Test-Driven Development (TDD)

    Development approach where tests are written before code (Red-Green-Refactor cycle). Ensures code meets requirements from the start.

    Cycle:

    1. Write failing test
    2. Write minimal code to pass
    3. Refactor code

    Behavior-Driven Development (BDD)

    Collaborative approach using natural language examples (Given-When-Then) to define requirements and automated tests.

    Example:

                    Given a user is logged in
                When they add an item to cart
                Then the cart should show 1 item

    Exploratory Testing

    Simultaneous learning, test design, and execution where testers explore the system without predefined scripts.

    Characteristics:

    • Session-based
    • Documented with charters
    • Time-boxed
    • Focused on discovery

    Continuous Testing

    Executing automated tests as part of the CI/CD pipeline to get immediate feedback on code changes.

    Key Components:

    • Unit tests (fast)
    • Integration tests
    • API tests
    • UI tests (selective)

    Agile Testing Metrics

    Test Automation Percentage

    Percentage of regression tests automated. Goal is high automation for repetitive tests.

    Defect Escape Rate

    Number of defects found in production versus earlier phases. Measures test effectiveness.

    Test Cycle Time

    Time from code commit to test completion. Should be minimized for fast feedback.

    Requirements Coverage

    Percentage of user stories with acceptance tests. Should be 100% for Done items.

    Defect Removal Efficiency

    Percentage of defects found before release. Higher is better for quality.

    Mean Time to Repair (MTTR)

    Average time to fix failed tests. Indicates team responsiveness.

    Challenges in Agile Testing

    1. Test Automation

    Creating and maintaining reliable automated tests requires significant investment and expertise.

    Solution: Start small, focus on high-value tests, and continuously improve automation skills.

    2. Changing Requirements

    Frequent changes can make tests obsolete quickly and require constant updates.

    Solution: Design flexible tests, focus on behavior over implementation, and collaborate closely with POs.

    3. Time Constraints

    Short iterations may pressure testing activities and lead to quality compromises.

    Solution: Shift-left testing, automate regression, and focus on risk-based testing.

    4. Cross-Functional Teams

    Testers may need broader skills beyond traditional testing expertise.

    Solution: Continuous learning, pair testing with developers, and skill diversification.
